# 项目状态评估

更新时间：2026-06-14

v1.0 完工标准以 `docs/V1_COMPLETION_STANDARD.md` 为准；本文只记录当前状态、限制和下一步。

## 总体判断

这个项目目前已经超过普通 MVP。它不是简单的“买入/卖出模拟器”，而是一个围绕新手训练设计的本地美股历史盲测系统：有真实/合成案例、强盲测、时间加速、下单纪律、仓位控制、资金结算、风险冷却、课程闸门、复盘报告、长期画像、错题补练和案例包导入导出。

我现在对它的评价是：

- 作为学习投资纪律的个人训练工具：已经很有价值。
- 作为真实交易前的训练场：方向正确，但还需要更多真实案例和更稳定的数据工作流。
- 作为券商级模拟交易系统：还不是，也暂时不应该往那个方向过度工程化。

## 已经做得好的地方

- 训练目标正确：重点不是预测最高点最低点，而是训练证据、仓位、计划、情绪和复盘。
- 盲测机制比较严肃：考试模式、随机盲测、提前揭晓污染、有效样本统计，都能减少“我其实已经知道答案”的假进步。
- 新手保护足够多：最大回撤、单仓上限、现金底线、禁止杠杆、T+1、good faith 类风险、滑点和流动性提醒都有覆盖。
- 复盘闭环很强：不是只看赚亏，而是看过程质量、错误根因、信心校准、反事实对比、任务完成和下一轮硬规则。
- 案例扩展路线已经打通：现在支持 CSV、完整 JSON 案例包、当前案例包导出，并有 `tools/build-case-package.mjs` 生成单个训练包，`tools/validate-case-package.mjs` 离线质检完整度，`tools/build-case-library.mjs` 按 manifest 批量构建和校验案例库，`tools/bundle-case-library.mjs` 把多个案例包打成可一次导入的课程 bundle，`tools/audit-case-coverage.mjs` 审计整套案例库是否偏科，`tools/scaffold-case-config.mjs` 按缺口生成案例配置模板。
- 网页端也能做案例库体检：左侧案例列表下方会即时检查当前内置和自定义案例库的覆盖面；当前 23 个内置训练包已经能达到结构均衡，考试模式会隐藏具体维度，避免污染盲测。
- starter 经典训练库已经成型并内置：`cases/starter-classic-library.json` 现在包含 11 个匿名经典形态案例，覆盖 ETF 系统性风险、快速反弹、单股事件、财报预期差、宏观利率、银行流动性、行业轮动、热门股追高、防御配置、公司行动、订单执行和 T+1 结算；覆盖审计达到 ready，并已生成 `packages/case-c-bundle.json` 和网页开箱加载用的 `data/starter-bundle.js`。重复导入同一个 starter bundle 会按内容指纹跳过。

## 现在仍然要保持清醒的地方

- 它不是实时交易系统。内置案例和 JSON/CSV 导入都是历史回放；生成器只是在运行时拉取历史 OHLCV，不会持续实时同步。
- 当前主要是日线训练，不适合训练分钟级、盘中盘口、期权、融资、做空或高频策略。
- 新闻不是自动全量抓取。现在更适合“人工精选经典事件 + 摘要 + 来源链接”的课程式训练，避免版权问题和信息噪音。
- 数据源需要核对。Yahoo 历史行情适合个人学习，但正式案例最好用多个来源交叉检查，尤其是拆股、分红、复权和停牌。
- 持久化仍偏个人项目。浏览器本地存储足够 MVP 使用，但长期大量训练最好后续加文件备份、版本化案例库或轻量数据库。
- `app.js` 体量仍然很大。虽然已经拆出大量核心逻辑和本轮计划中的四个 UI 模块，但事件处理、页面状态编排和部分渲染函数还在主文件里。后续继续维护时，应按低风险边界逐步迁出，不要为了追求“文件变小”一次性大改。

## 下一阶段最应该做的事

1. 建立经典案例库

   先不要追求实时行情。现在已经有 23 个内置训练包，其中 11 个匿名 starter 训练案例作为结构样板，已经越过 v1.0 的 20 个案例门槛。下一步最有价值的是继续把匿名经典形态替换或扩展为核对过的真实历史包，覆盖更细的单股黑天鹅、慢熊、快速反弹、ETF 防守失效和财报预期差变体。

2. 把案例生成流程标准化

   继续强化 `tools/build-case-package.mjs`、`tools/validate-case-package.mjs`、`tools/build-case-library.mjs`、`tools/audit-case-coverage.mjs` 和 `tools/scaffold-case-config.mjs`：让它们支持更多真实案例模板、来源清单模板、新闻/事件日期映射检查、覆盖审计和输出目录规范。这样项目会从“一个网页”变成“训练内容生产系统”。

3. 继续拆分前端代码

   `app.js` 已经很长，目前已经拆出了 `core/case-library-coverage.js`、`core/trade-ledger.js`、`core/orders.js`、`core/portfolio.js`、`core/course.js`、`core/runtime.js`、`core/review.js`、`core/importers.js`、`state/session.js`、`ui/training-plan.js`、`ui/profile-insights.js`、`ui/review-summary.js`、`ui/case-list.js`、`ui/risk-dashboard.js`、`ui/trade-form.js`、`ui/import-export.js` 和 `ui/events.js`。下一步不是盲目追求大重构，而是继续拆这些低风险边界：

   - `data/cases.js`：把内置案例数据和页面编排分开。
   - `ui/history.js`：迁出交易历史、公司行动和决策记录渲染。
   - `ui/course-panel.js`：迁出课程讲义、检查题和课程路径渲染。
   - `ui/profile-panel.js`：迁出长期画像入口和今日计划以外的画像概览。
   - `state/actions.js`：逐步收拢页面事件处理和状态更新。

   这样以后修 bug 和加功能不会变成在 60 多万字节的主文件里找针。

4. 继续强化真实使用保护

   实盘前准备度已经有硬门槛：至少 20 次有效随机盲测、最近 10 轮纪律均分不低于 80、平均回撤受控、不能连续破坏训练合约、风险冷却后不能违规买入、错题不能反复复发，并且交易机制、风险控制和下单执行都必须过底线。下一步不是放松门槛，而是继续把这些缺口和自动排课联动得更细。它只给训练建议，不给投资建议。

5. 继续强化浏览器端回归

   现在已有 `npm run browser:smoke` 覆盖首屏、按钮、案例库体检、导入案例包、文件选择导入、starter bundle 导入和重复跳过、备份导出恢复、随机盲测、课程检查题、真实观望记录、真实买入/卖出成交、限价单挂起/成交/过期/取消、持仓、资金和结算历史、导出下载、揭晓复盘、Markdown 下载和移动端宽度。下一步可以继续扩展到更多表单路径和异常输入路径。

## 给你的使用建议

你醒来后不要急着上实盘，也不要急着继续加花哨功能。先用随机盲测跑几轮，重点看自己会不会犯这些错误：

- 看到大跌就以为便宜。
- ETF 仓位开太大，以为分散就没有风险。
- 猜底猜顶，不写失效条件。
- 利好新闻出来后立刻追。
- 明明触发亏损线，还想再等等。
- 赚钱后不处理盈利计划，最后浮盈回吐。

如果这个工具能反复抓住这些问题，它就已经在做正确的事。
